## Overview

為 Koma 加入真實身分登入:**better-auth + Email 6-digit OTP**。不碰社群登入以閃過 Apple Guideline 4.8。登入後身分綁 `userId`、可跨裝置同步、未來可綁金流。現有「邀請碼 + HMAC cookie」系統與新系統**並存**,逐步淘汰;`access_codes` 保留當未來 beta gate(本期不新建閘門)。

## 決策摘要(已由 /evidence-check 定案)

- 實作方案 better-auth(唯一原生 Drizzle + SQLite/Turso;Lucia 作者背書;Auth.js v5 已維護模式)。Clerk / Supabase Auth / Lucia 排除。
- 只上 email OTP(非 magic link — magic link 的失效模式正是「桌機請求、手機點連結」的跨裝置情境,正中本案核心目標)。
- 不用 SMS OTP(NIST SP 800-63B-4 列 restricted)。
- 不碰社群登入 → 不觸發 Apple 4.8。passkey / Google / Apple 之後當加值,本期不做。

## 架構:雙系統並存模型

```
getServerAuth() 解析順序:
  1. better-auth session 有效 → { role, identity: `user:<userId>` }   ← 新,真實身分
       role = ADMIN_EMAILS 命中 ? "admin" : "member"
  2. 舊 HMAC koma_session 有效 → 既有行為(admin/member:<uuid>)        ← 過渡,最終移除
  3. 否則 → guest(hashed IP)                                          ← 不變
```

**額度綁定的關鍵 lazy 點**:`tts_usage.identity` 是自由字串、主鍵 `(identity, day)`。`user:<userId>` 只是**新命名空間**,與舊的 `member:` / `guest:` / `admin:` 共存 → **tts_usage 零 schema 變更、零資料遷移**。舊 member cookie 的桶沿用到他改用 email 登入為止。

**角色模型**:任何 email-verified 使用者 = member tier(每日 30 章,沿用 `TTS_QUOTA_MEMBER`);`ADMIN_EMAILS` env 允許清單 = admin(無限);未登入 = guest(5 章)。email OTP 取代邀請碼成為主要會員入口。

## Architecture Notes

**決策 1:better-auth 與舊 HMAC cookie 並存,而非一次性切換。**
- 排除的替代:(a) 直接砍舊系統 → 現有測試者 cookie 立即失效、需強制全員重登,風險高;(b) 把舊邀請碼資料遷移成 better-auth 帳號 → 邀請碼無 email、無法生帳號,遷移無意義。並存讓兩條路各自運作,自然淘汰。
- 相容性:`getServerAuth()` 是唯一身分入口,只改它一處;`tts_usage` / `tryConsumeQuota` / `getQuotaStatus` 全部不動(吃 `Auth.identity` 字串)。
- 可擴展:之後加 Google/Apple/passkey 都是 better-auth plugin,不動橋接層。

**決策 2:額度綁 `user:<userId>` 新命名空間,不遷移 tts_usage。**
- 排除的替代:加 `userId` 欄位 + 資料遷移 → 舊 identity 無對應 userId,遷移無從對應,且改主鍵風險高。新命名空間零遷移、零風險。
- 天花板(ponytail):舊 member 在改用 email 登入前,額度桶仍是舊 `member:<uuid>`;改登入後換到 `user:<id>` 桶(等於當日額度重置一次)。可接受。

**決策 3:email 供應商用 Resend(fetch,不加 SDK)。**
- 排除的替代:nodemailer + SMTP(要管 SMTP 設定);Resend SDK(單封信不需要)。
- 前置:Resend 需驗證寄件網域(外部設定,非程式碼)→ 列為風險。

**文件影響**:env 文件、部署 how-to、新增一頁雙系統 explanation、codemaps。
